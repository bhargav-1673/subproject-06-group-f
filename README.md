# C-DAC MPPLAB Internship 2026

## Sub-Project-06: Multi-Threaded Telecom Data Extraction Server

### Overview

This repository contains the implementation of **Sub-Project-06** under the C-DAC MPPLAB Summer Internship Program (June 2026).

The project focuses on the design and implementation of a **Multi-Threaded Streaming Media Player and Telecom Data Extraction Framework** using **POSIX Threads (pthreads)** in C.

The implementation simulates how packet data generated by multiple telecom sites is transferred to a centralized Data Extraction Server through a shared memory system. The architecture is designed to be generic enough to support both multimedia streaming data and telecom packet data.

---

## Project Objectives

* Implement a multi-threaded producer-consumer architecture using POSIX threads.
* Design a reusable and generic packet abstraction.
* Develop a thread-safe circular shared buffer.
* Synchronize concurrent access using POSIX Read-Write Locks.
* Simulate packet generation, buffering, and consumption.
* Provide a clean API that can later be integrated with MPI-based distributed systems.
* Support future expansion into hybrid MPI + Pthreads architectures.

---

## Telecom Scenario

The system models a real-world telecom environment where multiple city or regional sites periodically generate data packets and transmit them to a centralized server.

```text
Site-A ----\
Site-B -----\
Site-C ------> Data Extraction Server
Site-D -----/
Site-E ----/

                  |
                  v

        Shared Circular Buffer

                  |
                  v

             Consumers
```

Each packet is represented by a generic data structure and processed through a shared-memory producer-consumer pipeline.

# Shared Data Structure

The system uses a generic packet representation called `DataUnit`.

```c
typedef struct
{
    int packet_id;
    char source[32];
    char data[1024];
    int size;
    time_t timestamp;

} DataUnit;
```

### Design Goals

* Generic enough for telecom packets.
* Generic enough for media frames.
* Reusable by MPI integration modules.
* Independent of data source.

---

# Shared Buffer Design

The shared memory region is implemented as a circular queue.

```c
typedef struct
{
    DataUnit buffer[BUFFER_SIZE];

    int head;
    int tail;
    int count;

    pthread_rwlock_t rwlock;

} SharedBuffer;
```

### Features

* Fixed-size circular buffer.
* Thread-safe access.
* Configurable buffer size.
* Graceful handling of:

  * Buffer Full
  * Buffer Empty

---

# Public API

The shared buffer layer exposes the following API:

```c
void buffer_init(SharedBuffer *sb);

void buffer_destroy(SharedBuffer *sb);

int enqueue(
        SharedBuffer *sb,
        const DataUnit *packet);

int dequeue(
        SharedBuffer *sb,
        DataUnit *packet);
```

### Return Codes

```c
#define BUFFER_SUCCESS  0
#define BUFFER_FULL    -1
#define BUFFER_EMPTY   -2
```

---

# Synchronization Strategy

The project uses POSIX Read-Write Locks.

### Write Operations

The following operations modify shared state and therefore require:

```c
pthread_rwlock_wrlock()
```

* enqueue()
* dequeue()

### Read Operations

Read-only operations may use:

```c
pthread_rwlock_rdlock()
```

Example:

```c
peek()
```

This design ensures correctness when multiple producers and consumers are active simultaneously.

# Development Guidelines

### Important Rule

Direct access to the internal buffer array is prohibited.

Always use:

```c
enqueue()
dequeue()
```

Do not manipulate:

```c
head
tail
count
buffer[]
```

outside the shared buffer implementation.
