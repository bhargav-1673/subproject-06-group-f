## Overview
This repository contains the implementation of Sub-Project-06 under the C-DAC MPPLAB Summer Internship Program (June 2026). The project focuses on the design and implementation of a Multi-Threaded Streaming Media Player and Telecom Data Extraction Framework using POSIX Threads (pthreads) in C.

The implementation simulates how packet data generated by multiple telecom sites is transferred to a centralized Data Extraction Server through a shared memory system. The architecture is designed to be generic enough to support both multimedia streaming data and telecom packet data.

## Project Objectives
- Implement a multi-threaded producer-consumer architecture using POSIX threads.
- Design a reusable and generic packet abstraction.
- Develop a thread-safe circular shared buffer.
- Synchronize concurrent access using POSIX Read-Write Locks.
- Simulate packet generation, buffering, and consumption.
- Provide a clean API that can later be integrated with MPI-based distributed systems.
- Support future expansion into hybrid MPI + Pthreads architectures.

## Telecom Scenario
The system models a real-world telecom environment where multiple city or regional sites periodically generate data packets and transmit them to a centralized server.

```text
Site-A ----\
Site-B -----\
Site-C ------> Data Extraction Server
Site-D -----/
Site-E ----/

                  |
                  v

        Shared Circular Buffer

                  |
                  v

             Consumers
```
Each packet is represented by a generic data structure and processed through a shared-memory producer-consumer pipeline.

3. **Consumer Layer:** Dequeues and processes packets.
4. **Integration Layer:** Orchestrates thread creation, coordination, and lifecycle management.

## Shared Data Structure
The system uses a generic packet representation called `DataUnit`.
```c
typedef struct {
    int packet_id;
    char source[32];
    char data[1024];
    int size;
    time_t timestamp;
} DataUnit;
```
**Design Goals:** Generic enough for telecom packets and media frames, reusable by MPI integration modules, and independent of the data source.

## Shared Buffer Design
The shared memory region is implemented as a circular queue.
```c
typedef struct {
    DataUnit slots[BUFFER_SIZE];
    int head;
    int tail;
    int count;
    pthread_rwlock_t lock;
} SharedBuffer;
```
**Features:**
- Fixed-size circular buffer.
- Thread-safe access.
- Configurable buffer size.
- Graceful handling of Buffer Full and Buffer Empty conditions.

### Public API
```c
void buffer_init(SharedBuffer *sb);
void buffer_destroy(SharedBuffer *sb);
int enqueue(SharedBuffer *sb, const DataUnit *packet);
int dequeue(SharedBuffer *sb, DataUnit *packet);
```

### Return Codes
```c
#define BUF_OK     0
#define BUF_FULL  -1
#define BUF_EMPTY -2
#define BUF_ERR   -3
```

## Synchronization Strategy
The project uses POSIX Read-Write Locks to ensure correctness when multiple producers and consumers are active simultaneously.

| Function    | Lock Type  | Reason                          |
|-------------|------------|---------------------------------|
| `enqueue()` | WRITE Lock | Modifies `tail` and `count`     |
| `dequeue()` | WRITE Lock | Modifies `head` and `count`     |

*Note:* `pthread_rwlock_rdlock()` is reserved for future read-only operations such as `peek()` or buffer status queries. The rwlock is chosen over a plain mutex specifically to support this extension point.

## Development Guidelines

### Important Rule: Shared Buffer API
**Never access `SharedBuffer` internals directly.**
- ❌ **Incorrect:** `buf->slots[i]`, `buf->count`, `buf->head`, `buf->tail`
- ✅ **Correct:** `enqueue(&buffer, &packet);`, `dequeue(&buffer, &packet);`

Only the buffer module should manipulate internal buffer state.
