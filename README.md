## Overview
This repository contains the implementation of Sub-Project-06 under the C-DAC MPPLAB Summer Internship Program (June 2026). The project focuses on the design and implementation of a Multi-Threaded Streaming Media Player and Telecom Data Extraction Framework using POSIX Threads (pthreads) in C.

The implementation simulates how packet data generated by multiple telecom sites is transferred to a centralized Data Extraction Server through a shared memory system. The architecture is designed to be generic enough to support both multimedia streaming data and telecom packet data.

## Project Objectives
- Implement a multi-threaded producer-consumer architecture using POSIX threads.
- Design a reusable and generic packet abstraction.
- Develop a thread-safe circular shared buffer.
- Synchronize concurrent access using POSIX Read-Write Locks.
- Simulate packet generation, buffering, and consumption.
- Provide a clean API that can later be integrated with MPI-based distributed systems.
- Support future expansion into hybrid MPI + Pthreads architectures.

## Telecom Scenario
The system models a real-world telecom environment where multiple city or regional sites periodically generate data packets and transmit them to a centralized server.

```text
Site-A ----\
Site-B -----\
Site-C ------> Data Extraction Server
Site-D -----/
Site-E ----/

                  |
                  v

        Shared Circular Buffer

                  |
                  v

             Consumers
```
Each packet is represented by a generic data structure and processed through a shared-memory producer-consumer pipeline.

## Technical Requirements
- **Language:** C (ISO C Standard)
- **Operating System:** Ubuntu Linux
- **Compiler:** `gcc -O2 -pthread`
- **Thread Library:** POSIX Threads (pthreads)
- **Synchronization:** `pthread_rwlock_t` (`pthread_rwlock_init`, `pthread_rwlock_rdlock`, `pthread_rwlock_wrlock`, `pthread_rwlock_unlock`, `pthread_rwlock_destroy`)

## Core Components
1. **Producer Layer:** Generates packets and enqueues them into the shared buffer.
2. **Shared Buffer Layer:** Thread-safe circular queue managing data flow between producers and consumers.
3. **Consumer Layer:** Dequeues and processes packets concurrently.
4. **Integration Layer:** Orchestrates thread creation, coordination, and lifecycle management via `media_player.c`.
5. **Metrics/Telemetry Layer:** Thread-safe runtime instrumentation tracking throughput, latency, buffer occupancy, packet loss, and per-thread producer/consumer statistics.

## Shared Data Structure
The system uses a generic packet representation called `DataUnit`.
```c
typedef struct {
    int packet_id;
    char source[32];
    char data[1024];
    int size;
    time_t timestamp;
} DataUnit;
```
**Design Goals:** Generic enough for telecom packets and media frames, reusable by MPI integration modules, and independent of the data source.

## Shared Buffer Design
The shared memory region is implemented as a circular queue.
```c
typedef struct {
    DataUnit slots[BUFFER_SIZE];
    int head;
    int tail;
    int count;
    pthread_rwlock_t lock;
} SharedBuffer;
```
**Features:**
- Fixed-size circular buffer.
- Thread-safe access.
- Configurable buffer size.
- Graceful handling of Buffer Full and Buffer Empty conditions.

### Public API
```c
void buffer_init(SharedBuffer *sb);
void buffer_destroy(SharedBuffer *sb);
int enqueue(SharedBuffer *sb, const DataUnit *packet);
int dequeue(SharedBuffer *sb, DataUnit *packet);
```

### Return Codes
```c
#define BUF_OK     0
#define BUF_FULL  -1
#define BUF_EMPTY -2
#define BUF_ERR   -3
```

## Synchronization Strategy
The project uses POSIX Read-Write Locks to ensure correctness when multiple producers and consumers are active simultaneously.

| Function    | Lock Type  | Reason                          |
|-------------|------------|---------------------------------|
| `enqueue()` | WRITE Lock | Modifies `tail` and `count`     |
| `dequeue()` | WRITE Lock | Modifies `head` and `count`     |

*Note:* `pthread_rwlock_rdlock()` is reserved for future read-only operations such as `peek()` or buffer status queries. The rwlock is chosen over a plain mutex specifically to support this extension point.

## Metrics and Telemetry

The system includes a dedicated thread-safe metrics module (`metrics.h` / `metrics.c`) that instruments the pipeline at runtime and prints a structured telemetry report at the end of each integration run.

### MetricsTracker Structure
```c
typedef struct MetricsTracker {
    pthread_mutex_t lock;
    struct timespec start_time;
    struct timespec end_time;

    int packets_generated;
    int packets_enqueued;
    int packets_dequeued;
    int packets_dropped;

    long long total_latency_ms;
    long long occupancy_sum;
    int occupancy_samples;
    int max_occupancy;

    /* Per-thread stats (0-based index, MAX_THREADS=16) */
    int producer_enqueued[MAX_THREADS];
    int producer_dropped[MAX_THREADS];
    int consumer_processed[MAX_THREADS];
} MetricsTracker;
```

### Public API
```c
void metrics_init(MetricsTracker *tracker);
void metrics_start(MetricsTracker *tracker);
void metrics_stop(MetricsTracker *tracker);

/* Aggregate variants — used by standalone buffer/producer tests */
void metrics_log_enqueue(MetricsTracker *tracker, int count);
void metrics_log_drop(MetricsTracker *tracker);
void metrics_log_dequeue(MetricsTracker *tracker, int count, time_t packet_timestamp);

/* Per-thread variants — used by integration layer */
void metrics_log_enqueue_by(MetricsTracker *tracker, int count, int producer_id);
void metrics_log_drop_by(MetricsTracker *tracker, int producer_id);
void metrics_log_dequeue_by(MetricsTracker *tracker, int count, time_t packet_timestamp, int consumer_id);

void metrics_print_report(const MetricsTracker *tracker, int num_producers, int num_consumers, int expected_packets);
```

## Development Guidelines

### Important Rule: Shared Buffer API
**Never access `SharedBuffer` internals directly.**
- ❌ **Incorrect:** `buf->slots[i]`, `buf->count`, `buf->head`, `buf->tail`
- ✅ **Correct:** `enqueue(&buffer, &packet);`, `dequeue(&buffer, &packet);`

Only the buffer module should manipulate internal buffer state.

### Branch Strategy
```text
main
├── feature/shared_buffer
├── feature/producer_module
├── feature/consumer_module
├── feature/integration
└── documentation
```
